# Minimal compiler constant-time harness

This directory is a small, standalone corpus for inspecting compiler output
from C.  It deliberately does **not** depend on the AISec MLIR dialect or any
project pass yet.  The C files are reduced reproductions; the MLIR files use
only standard MLIR/LLVM dialects and carry comments explaining where a secret
becomes observable.

The sources are defensive regression tests, not key-recovery programs.

## Build with the installed Homebrew LLVM

```sh
make -C c verify
make -C c regen-mlir
# Optional per-case builds:
make -C c clangover
make -C c wolfssl-3580
make -C c wolfssl-3579
make -C c kyberslash
```

The Makefile uses `/opt/homebrew/opt/llvm/bin` by default.  Override `LLVM_BIN`
when using another LLVM installation.  Cross-target assembly is produced for
x86-64, RV32I, and AArch64.  The current machine does not include GCC's RISC-V
toolchain, so GCC-specific wolfSSL reproductions are marked as unavailable
unless `RISCV_GCC` is supplied.

## Cases

* `clangover_poly_frommsg_*`: Clang's select/mask-to-branch behavior described
  by [Clangover](https://github.com/antoonpurnal/clangover) and
  [PQShield](https://pqshield.com/pqshield-plugs-timing-leaks-in-kyber-ml-kem-to-improve-pqc-implementation-maturity/).
* `wolfssl_3580_*`: a reduced fixed-count table selection from
  [CVE-2026-3580](https://osv.dev/vulnerability/CVE-2026-3580).
* `wolfssl_3579_*`: a 64-bit multiply that becomes `__muldi3` on RV32I, from
  [CVE-2026-3579](https://nvd.nist.gov/vuln/detail/CVE-2026-3579).
* `kyberslash1_*` and `kyberslash2_*`: the two constant-divisor kernels from
  [KyberSlash](https://kyberslash.cr.yp.to/kyberslash-20240628.pdf), with the
  official fixes recorded in `c/` comments.

The checked-in MLIR is an annotated, review-sized snapshot of the decisive
operations.  `build/mlir/` is the exact output of the importer and is ignored
by Git; it is regenerated by `make -C c regen-mlir`.  For backend-only
phenomena, the MLIR comment says `MODELED FROM ASSEMBLY`; LLVM's importer
cannot see a GCC backend decision that occurs after LLVM IR.

## Observations from this machine

With Homebrew LLVM 17.0.6, the reductions currently show:

| reduction | vulnerable observation | fixed observation |
| --- | --- | --- |
| Clangover | x86 assembly contains `btl ...; jae ...` | caller uses `clangover_ct_cmov`; no `jae` |
| wolfSSL 3580 | GCC/RV32I model contains secret-derived `bnez` | fixed LLVM reduction is mask/select based |
| wolfSSL 3579 | RV32I assembly calls `__muldi3@plt` | fixed assembly has a public 64-iteration loop |
| KyberSlash 1/2 | imported LLVM dialect contains `llvm.udiv` | imported LLVM dialect contains multiply/shift only |

The local machine has no RISC-V GCC, so the 3580 GCC assembly check prints a
clear `SKIP`.  Supplying `RISCV_GCC=/path/to/riscv32-gcc` enables that check;
the checked-in MLIR remains a target-level model until then.

There is no key-recovery code here.  The future SPS integration can consume
the comments and the vulnerable/fixed pairs as lit-style security harnesses.
